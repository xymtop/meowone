# A2A 协议介绍

## 概述

**A2A（Agent2Agent）协议**是由 Google 于 2025 年 4 月推出的开放标准协议，旨在实现 AI 智能体（Agent）之间的互联互通与协作。目前该协议已捐赠给 Linux 基金会托管，作为开源项目持续发展。

### 背景与动机

在当前的 AI 应用场景中，不同组织、使用不同技术栈开发的 AI 智能体往往处于"信息孤岛"状态，难以有效沟通和协作。A2A 协议正是为了解决这一核心挑战而生——它充当智能体生态系统中的"通用语言"或"翻译器"，打破数据孤岛，增强智能体间的互操作性。

想象一个实际场景：用户请求 AI 助手规划一次国际旅行，这可能需要协调多个专业智能体：
- 航班预订智能体 - 处理机票搜索和预订
- 酒店预订智能体 - 管理住宿安排
- 当地旅游智能体 - 提供景点推荐
- 金融服务智能体 - 处理货币转换

没有通用通信协议的情况下，集成这些多样化智能体需要大量定制化解决方案，系统难以扩展和维护。

## 核心架构组件

A2A 协议由以下几个关键构建块组成：

### 1. A2A 客户端（Client Agent）

也称为"客户端智能体"，是发起请求的应用程序、服务或其他 AI 智能体。它使用 A2A 协议向远程智能体发起通信请求。

### 2. A2A 服务器（Server Agent）

也称为"远程智能体"，负责接收请求、处理任务并通过状态更新或结果进行响应。它公开一个与 A2A 协议兼容的 HTTP 端点。

### 3. Agent Card（智能体名片）

Agent Card 是 A2A 协议的核心创新之一，它是一个 JSON 格式的元数据文件，通常可通过 `/.well-known/agent.json` 路径访问。Agent Card 包含：

| 字段 | 说明 |
|------|------|
| `name` | 智能体名称 |
| `description` | 智能体功能描述 |
| `url` | 服务端点 URL |
| `version` | 协议版本 |
| `capabilities` | 支持的能力（如 streaming、pushNotifications） |
| `skills` | 具体技能列表 |
| `authentication` | 认证要求 |

Agent Card 类似于 AI 领域的"模型卡"（Model Card），相当于智能体的"名片"或"简历"，使智能体之间能够相互发现和了解彼此的能力。

### 4. Task（任务）

任务代表完成请求所需的工作单元，具有以下特征：
- 每个任务拥有智能体定义的唯一 ID
- 具有完整的生命周期状态：已提交（submitted）、处理中（working）、需要输入（input-required）、已完成（completed）、失败（failed）
- 支持多轮交互和长期运行操作

### 5. Message（消息）

消息是通信的基本单元，表示一次对话中的单次交换。消息包含：
- **角色**：`"user"`（客户端发送）或 `"agent"`（服务端发送）
- **内容部分**：包含一个或多个 Part 对象

### 6. Part（内容部分）

Part 是消息或工件内部的内容载体，分为三种类型：

| Part 类型 | 用途 | 示例 |
|-----------|------|------|
| `TextPart` | 纯文本内容 | 指令、问题、回答 |
| `FilePart` | 文件传输 | 文档、图片、数据文件 |
| `DataPart` | 结构化数据 | JSON 表单、参数 |

### 7. Artifact（工件）

工件是任务执行过程中生成的有形交付成果，如文档、图像、电子表格等。工件可以增量流式传输。

## 技术实现

### 通信基础

| 技术要素 | 说明 |
|----------|------|
| **传输协议** | HTTP/HTTPS |
| **数据格式** | JSON-RPC 2.0 |
| **实时通信** | Server-Sent Events (SSE) |
| **身份验证** | OAuth 2.0、API Keys、Bearer Token |

### 交互机制

A2A 协议支持三种交互模式，以适应不同的应用场景：

#### 1. 同步请求/响应
- **适用场景**：简单查询、快速任务
- **实现方式**：HTTP 请求 + 轮询
- **特点**：简单直接，但效率相对较低

#### 2. 流式传输（Streaming）
- **适用场景**：实时更新、增量结果
- **实现方式**：Server-Sent Events (SSE)
- **特点**：实时性好，但需要持续连接

#### 3. 推送通知（Push Notifications）
- **适用场景**：长期任务、异步处理
- **实现方式**：Webhook 回调
- **特点**：适合长时间运行的任务，实现相对复杂

### 工作流程

A2A 协议遵循三步工作流程：

```
┌─────────────────────────────────────────────────────────┐
│                     1. 发现 (Discover)                   │
│  客户端智能体查找远程智能体，获取 Agent Card 确定能力     │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                   2. 认证 (Authenticate)                 │
│  根据 Agent Card 中的安全方案进行身份验证               │
│  支持 OAuth 2.0、API Keys、OpenID Connect 等            │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                   3. 通信 (Communicate)                 │
│  客户端向远程智能体发送任务，通过 JSON-RPC 交互          │
│  支持流式传输和异步更新                                  │
└─────────────────────────────────────────────────────────┘
```

## A2A 与 MCP 的区别

A2A 协议与 MCP（Model Context Protocol）协议经常被一起提及，它们既有区别又互为补充：

| 对比维度 | A2A 协议 | MCP 协议 |
|----------|----------|----------|
| **主要用途** | AI 智能体间对等协作 | AI 模型与工具/资源连接 |
| **交互特点** | 状态化、多轮对话、协商式 | 无状态、单次调用、事务性 |
| **应用场景** | 智能体委托、协作项目管理 | 函数调用、API 查询、数据获取 |
| **复杂度** | 支持复杂、动态交互 | 结构化、可预测的输入输出 |
| **设计目标** | 水平协作（Agent ↔ Agent） | 垂直集成（Agent → Tools） |

### 协作示例：汽车维修店

```
                    客户
                      │
                      │ A2A
                      ↓
              ┌───────────────┐
              │   店长智能体   │
              └───────┬───────┘
                      │ A2A
        ┌─────────────┼─────────────┐
        ↓             ↓             ↓
   ┌─────────┐   ┌─────────┐   ┌───────────┐
   │机械师智能体│   │  MCP    │   │供应商智能体│
   └────┬────┘   │ 连接器   │   └───────────┘
        │        │         │
        ├───────→│诊断扫描仪│
        ├───────→│维修手册  │
        └───────→│升降平台  │
```

**场景分析：**
1. **客户交互（A2A）**：客户通过 A2A 与店长智能体多轮对话诊断问题
2. **内部工具使用（MCP）**：机械师智能体使用 MCP 调用专业诊断工具
3. **供应商协作（A2A）**：机械师智能体与供应商智能体协商零件采购

## A2A 协议的优势

### 1. 隐私保护
A2A 将智能体视为"不透明"单元，智能体可以在不暴露内部工作原理的情况下进行协作，包括：
- 内部存储器
- 专有逻辑
- 特定工具实现

这有助于保护数据隐私和知识产权。

### 2. 无缝集成
A2A 建立在成熟的标准之上：
- HTTP/HTTPS
- JSON-RPC 2.0
- SSE（Server-Sent Events）

这使得企业更容易采用该协议，并确保与现有技术栈的兼容性。

### 3. 安全性
协议内置企业级安全特性：
- 支持 OAuth 2.0、API Keys、Bearer Token 等认证方案
- HTTPS 加密传输
- 授权和访问控制机制

### 4. 互操作性
超过 50 家技术合作伙伴支持 A2A 协议，包括：
- Atlassian、Box、Cohere
- MongoDB、PayPal、Salesforce
- SAP、ServiceNow、Workday

### 5. 多语言 SDK 支持
官方提供多种编程语言的 SDK：
- Python
- JavaScript/TypeScript
- Java
- Go
- .NET

## 实际应用场景

### 场景一：国际旅行规划

```
用户 ──→ 主助手 ──→ 航班智能体
              │
              ├──→ 酒店智能体
              │
              └──→ 旅游智能体
```

用户向主助手提出旅行规划请求后，主助手协调多个专业智能体协作完成：
1. 查询航班选项并预订
2. 预订酒店住宿
3. 安排当地活动和行程

### 场景二：企业客服协作

多智能体协作处理客户服务请求：
1. **一线客服智能体** - 处理常见问题
2. **专业技术智能体** - 解决技术难题
3. **账单处理智能体** - 处理财务相关问题
4. **升级管理智能体** - 处理投诉和特殊情况

### 场景三：零售库存管理

```
库存智能体 ──→ 订单智能体 ──→ 供应商智能体
   │              │                │
   MCP            │                │
   ↓              ↓                ↓
数据库          订单系统         供应商API
```

库存智能体检测到产品库存不足时，通过 A2A 与订单智能体通信，订单智能体再与外部供应商智能体协商下单。

## 智能体发现机制

A2A 协议支持三种发现机制：

### 1. 标准 URI 发现（推荐）

```
https://{智能体服务器域名}/.well-known/agent.json
```

客户端向标准路径发送 HTTP GET 请求，服务器返回 Agent Card JSON 响应。

**优势**：简单、标准化，支持自动化发现

### 2. 策划注册表（目录式发现）

适用于企业环境和专业市场：
- 集中管理和治理
- 基于功能能力的发现
- 访问控制和信任机制

### 3. 直接配置（私有发现）

适用于：
- 紧密耦合系统
- 私有智能体
- 开发测试环境

## 安全最佳实践

| 安全层面 | 实施建议 | 技术方案 |
|----------|----------|----------|
| **认证** | 使用标准 Web 认证 | OAuth 2.0、Bearer Token |
| **授权** | 基于角色的访问控制 | JWT 令牌、权限矩阵 |
| **传输安全** | 强制 HTTPS | TLS 1.2+、证书验证 |
| **网络隔离** | 限制访问范围 | VPC、IP 白名单 |

## 未来发展方向

A2A 协议仍在快速发展中，未来规划包括：

1. **智能体卡增强**：加入授权方案和可选凭据
2. **动态技能检查**：检查意外或不受支持的技能的方法
3. **用户体验协商**：支持任务中的动态 UX（如在对话中添加音频或视频）
4. **增强推送通知**：改进通知方式和流式传输的可靠性

## 总结

A2A 协议代表了 AI 智能体生态系统发展的重要里程碑，它：

- ✅ 解决了智能体间协作的标准化问题
- ✅ 支持复杂的多轮对话和任务协作
- ✅ 提供企业级的安全性和隐私保护
- ✅ 与 MCP 协议互补，共同构建完整的智能体生态
- ✅ 由 Linux 基金会托管，确保开放和社区驱动

对于希望构建分布式、模块化 AI 智能体系统的开发者来说，A2A 协议是值得关注和投资的关键基础设施。
