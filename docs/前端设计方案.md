# 前端设计方案

## 1. 背景与问题

当前 `nextadmin` 已有较完整页面，但整体体验仍偏“后台配置台”：

- 页面能力分散：`sessions/messages/chat/models/mcp/skills/agents/internal-agents/scheduled-tasks/gateway-logs`
- 部分页面功能重叠（如 `agents` 与 `internal-agents`）
- 缺少统一任务闭环：从配置能力到实际运行、观测、回溯不连续
- 用户心智是“系统在工作”，当前呈现是“多个独立 CRUD 页面”

目标是形成“像系统一样”的整体体验：

1. 用户能明确看到系统当前状态（模型、能力、智能体、任务、运行日志）。  
2. 用户能沿主流程操作：配置 -> 运行 -> 观测 -> 调优。  
3. 页面间上下文连续，减少割裂感与重复入口。  

---

## 2. 现状盘点（基于已有代码）

已有核心页面：

- 会话中心：`/meowone/chat`、`/meowone/sessions`、`/meowone/messages`
- 能力中心：`/meowone/models`、`/meowone/capabilities-mcp`、`/meowone/capabilities-skills`、`/meowone/config`（prompt）
- 智能体与任务：`/meowone/agents`、`/meowone/internal-agents`、`/meowone/scheduled-tasks`
- 运维观测：`/meowone/gateway-logs`
- 测试页：`/meowone/agent-dag-test`、`/meowone/monaco-test`

关键问题：

1. `agents`（DB 管理）与 `internal-agents`（运行时管理）同时存在，用户不清楚差异。  
2. 会话页与配置页割裂，没有“系统状态总览”。  
3. 缺少“智能体详情页”，无法在一个地方查看：配置、资源绑定、运行结果、最近错误。  
4. 缺少任务运行看板，`scheduled-tasks` 与 `gateway-logs` 未打通。  

---

## 3. 信息架构（IA）重组

建议将导航改为四大域，保持现有路由兼容并逐步迁移：

1. **工作台（Workbench）**
   - `系统总览`（新）
   - `对话控制台`（复用 `/meowone/chat`）
2. **智能体中心（Agents）**
   - `智能体列表`（整合 `/meowone/agents`）
   - `智能体详情`（新，配置+运行+日志）
   - `任务调度`（复用 `/meowone/scheduled-tasks`）
3. **能力中心（Capabilities）**
   - `模型管理`（`/meowone/models`）
   - `MCP 管理`（`/meowone/capabilities-mcp`）
   - `技能管理`（`/meowone/capabilities-skills`）
   - `提示词管理`（`/meowone/config`）
4. **观测中心（Observability）**
   - `会话记录`（整合 `sessions/messages`）
   - `网关日志`（`/meowone/gateway-logs`）
   - `执行日志`（新，后端补接口后接入）

“测试”类页面移动到隐藏开发入口，不作为常规用户导航。

## 4. 关键页面设计

## 4.1 系统总览页（新）

目的：让用户一眼判断系统是否“可用、可调度、可追踪”。

模块：

- 系统健康：后端健康、默认模型、MCP 数量、启用技能数
- 智能体状态：内部/外部 agent 数量，最近失败 agent
- 调度状态：启用任务数、下一次执行、最近执行结果
- 运行入口：快速进入“新建智能体”“进入聊天”“查看日志”

数据来源（可由现有 API 拼装）：

- `/health`
- `/api/models`
- `/api/capabilities/mcp`
- `/api/capabilities/skills`
- `/api/agents`
- `/api/scheduled-tasks`
- `/api/gateway/logs`（近期错误事件）

## 4.2 智能体列表页（重构 `/meowone/agents`）

从“原始 JSON 列表”升级为“业务列表”：

- 列字段：名称、类型、协议、绑定资源摘要、状态、最近调用结果
- 支持筛选：internal/external、enabled、has_error
- 支持操作：编辑、复制、删除、进入详情、快速运行

编辑抽屉分两块：

1. 基础信息：name/type/description  
2. 资源绑定：prompt、skills、mcp、tools policy、limits  
3. external 专属：protocol(a2a)、base_url、timeout

## 4.3 智能体详情页（新）

这是“系统化体验”核心页，解决当前割裂。

结构：

- Tab1 配置：展示并编辑 agent 定义
- Tab2 调试运行：输入 task，实时展示执行结果
- Tab3 调度任务：绑定/查看该 agent 的 scheduled tasks
- Tab4 运行日志：最近执行历史与错误

效果：用户在一个页面完成“定义-验证-上线-监控”闭环。

## 4.4 对话控制台（优化 `/meowone/chat`）

保持现有优秀体验（SSE、tools、A2UI、Mermaid），增加系统感：

- 顶部显示当前默认模型、调度模式、渠道
- 侧边增加“快捷切换智能体模板”（可选）
- 每条 assistant 消息可展开查看“本轮工具调用摘要”

## 4.5 会话记录（整合 sessions/messages）

将 `sessions` 和 `messages` 合并成一个两栏页：

- 左栏：会话列表 + 搜索
- 右栏：消息时间线 + 元数据（role/content_type/card）
- 操作：删除、导出、复制 session id

## 5. 交互主流程设计（统一体验）

## 流程 A：新用户配置到可运行

1. 进入系统总览（看到“未配置默认模型”提醒）  
2. 跳转模型管理，设置默认模型  
3. 跳转能力中心配置 MCP/Skill/Prompt  
4. 在智能体中心创建内部 agent（绑定资源）  
5. 在详情页调试运行，通过后创建定时任务  
6. 在观测中心查看运行日志

## 流程 B：生产问题排查

1. 总览页看到“任务失败告警”  
2. 点击进入智能体详情日志 Tab  
3. 查看失败请求与工具调用  
4. 回到配置 Tab 修正 prompt/tool policy  
5. 在调试 Tab 重放任务确认修复

## 6. 组件与代码层落地建议

## 6.1 共享组件

新增共享 UI 组件（减少重复写法）：

- `ResourceBindingForm`（prompt/skills/mcp/tools/limits）
- `AgentTypeBadge`（internal/external/a2a）
- `ExecutionResultPanel`（结果、耗时、错误码）
- `HealthCards`（总览指标卡）
- `JsonPreviewDialog`（替代页面内大段 `<pre>`）

## 6.2 API 层封装

`meowone-api.ts` 保持单入口，补充：

- `getAgentDetail(name,type)`
- `executeAgent(...)`（若后端新增统一执行接口）
- `listAgentExecutionLogs(...)`

并统一错误处理：

- 标准化 toast/message 文案
- 对 400/404/500 给出用户可操作提示

## 6.3 状态管理

目前局部 `useState` 足够，但建议引入轻量全局 store（Zustand）管理：

- 当前工作上下文（选中 agent/session）
- 总览页聚合指标缓存
- 轮询状态（logs/tasks）

## 7. 导航与路由迁移方案

## 7.1 导航分阶段

阶段 1：

- 仅调整 sidebar 分组文案与排序（不改路由）

阶段 2：

- 新增 `系统总览` 和 `智能体详情`
- `internal-agents` 标记为“兼容页”，在 `agents` 内提供跳转提示

阶段 3：

- 将 `sessions/messages` 合并为 `会话记录`
- 旧页保留但从主导航移除

## 8. 页面优先级与实施计划

## P0（必须）

1. 导航重组（文案与分组）  
2. `agents` 页面重构为业务列表（去原始 JSON 暴露）  
3. 新增 `系统总览`（拼现有接口即可）  
4. `chat` 页顶部增加系统上下文信息（模型/模式）

## P1（高优先）

1. 新增 `智能体详情`（配置+调试+任务）  
2. 合并 `sessions/messages` 体验  
3. 统一错误提示与提交反馈

## P2（增强）

1. 观测中心补“执行日志”  
2. 支持从日志一键回放任务  
3. 布局与主题细节优化（系统感强化）

---

## 9. 视觉与文案规范建议

1. 页标题统一“领域名 + 动词”：如“智能体管理”“任务调度”。  
2. 强调系统状态：使用状态色（健康/告警/失败）而不是仅文本。  
3. 降低技术噪音：默认隐藏 JSON，提供“查看原始数据”按钮。  
4. 关键操作可追踪：保存后明确提示“已生效于哪些运行路径”。  

---

## 10. 验收标准（用户视角）

满足以下条件即认为“从割裂走向系统化”：

1. 用户能在 3 步内找到“配置智能体并运行”的完整路径。  
2. 任一运行异常都能在 2 个页面内定位（详情或观测中心）。  
3. 用户不需要理解内部/外部实现细节，也能完成主要操作。  
4. 不再依赖原始 JSON 表格才能完成管理任务。  

---

## 11. 给前端编程 Agent 的执行清单

1. 重构 `sidebar data`，按 Workbench/Agents/Capabilities/Observability 分组。  
2. 新建 `系统总览` 页面，聚合现有 API 指标卡。  
3. 重构 `agents/page.tsx`：列表化、表单化、删除原始 JSON 直出。  
4. 新建 `agents/[type]/[name]` 详情页（先做配置+调试两个 Tab）。  
5. `chat/page.tsx` 增加顶部系统上下文与工具调用摘要折叠。  
6. 合并 `sessions/messages` 为统一会话记录页（旧页保留兼容）。  
7. 统一弹窗与错误提示组件，替换 `confirm` + 零散错误文本。  

---

## 12. 结论

当前前端并不是能力不足，而是“系统链路未被产品化表达”。  
本方案不要求大规模重写，重点是：

- 统一信息架构
- 打通关键流程
- 建立“智能体详情页”作为系统中枢

这样用户从“点很多页面做配置”变成“在一个系统里完成任务闭环”。

## 13. 页面布局设计（需重点执行）

本节定义统一布局骨架，避免“每页一个样式语言”造成割裂。

### 13.1 全局布局骨架

统一采用三段式：

1. **Global Header（全局头部）**
   - 左：当前模块标题 + 面包屑
   - 中：全局搜索（会话、智能体、任务）
   - 右：系统状态灯（模型/MCP/任务健康）、用户菜单
2. **Left Sidebar（导航）**
   - 按 Workbench/Agents/Capabilities/Observability 分组
   - 支持折叠，仅保留图标与 hover 文案
3. **Main Content（主内容区）**
   - 页面头（标题、说明、关键 CTA）
   - 筛选与操作条（Filter Bar）
   - 主信息区（列表/详情/图表）

### 13.2 页面宽度与栅格

- 内容最大宽度建议 `1280px`（超宽屏避免信息过散）
- 通用栅格：
  - 桌面：12 栏
  - 平板：8 栏
  - 移动：4 栏
- 详情页结构建议：
  - 左 8 栏：主编辑/日志内容
  - 右 4 栏：状态摘要、操作按钮、关联信息

### 13.3 视觉层级规范

- 一级信息：页面标题、关键状态、主 CTA
- 二级信息：筛选条件、统计摘要、标签
- 三级信息：调试字段、原始 JSON、辅助说明

默认展示一级+二级，三级信息折叠，避免信息噪音。

---

## 14. 用户操作设计（核心路径）

## 14.1 智能体创建与上线路径（最关键）

将创建流程做成 Stepper（分步）而不是一页堆字段：

1. **基础信息**：名称、类型（internal/external）、描述  
2. **资源绑定**：prompt、skills、mcp、工具策略  
3. **执行参数**：limits、超时、（external: a2a 地址）  
4. **调试运行**：输入 task，实时返回结果  
5. **发布启用**：保存并可选创建定时任务

交互要求：

- 每步可保存草稿
- 显示“必填完成度”
- 下一步前做前端校验（例如 external 必须有 a2a base_url）

## 14.2 任务调度路径

从“填写表单”升级为“任务向导”：

1. 选择智能体
2. 输入任务模板
3. 配置周期/模式
4. 试运行一次（dry-run）
5. 启用任务

并在结果页展示：

- 下次执行时间
- 最近执行状态
- 最近错误摘要

## 14.3 异常排查路径

从任意失败态可一键跳转：

- 任务失败卡片 -> 智能体详情日志 Tab
- 日志行 -> 对应会话消息
- 会话消息 -> 工具调用明细

目标：用户不靠猜测，3 步内定位问题。

## 15. 信息展示设计（展示什么、怎么展示）

## 15.1 列表页信息密度策略

列表默认只显示“决策必需字段”：

- 智能体列表：名称、类型、状态、绑定摘要、最近运行时间
- 任务列表：名称、agent、周期、状态、next_run、last_status
- 会话列表：标题、最近更新时间、消息数、来源 channel

“原始数据 JSON”统一放入二级抽屉或详情页，不在主表直接铺开。

## 15.2 详情页信息架构（统一模板）

每个详情页都包含 4 个固定区域：

1. **Header Summary**：对象名、状态、最近变更时间、关键操作按钮  
2. **Configuration**：可编辑配置项（分组）  
3. **Runtime**：最近执行结果、耗时、轮次、错误码  
4. **History**：变更记录/执行记录/关联资源

好处是跨页面一致，用户学习成本低。

## 15.3 指标与状态展示

系统状态尽量“图形化 + 颜色语义一致”：

- 绿色：健康
- 黄色：告警（可运行但有风险）
- 红色：失败
- 灰色：未配置/未知

关键指标建议：

- 智能体：总数、启用数、最近 24h 失败数
- 任务：启用任务数、待执行任务数、平均执行耗时
- 对话：今日会话数、平均响应时长、tool error 比例

## 16. 关键页面线框建议（文字版）

## 16.1 系统总览

- 第一屏：4 张状态卡（模型/MCP/智能体/任务）
- 第二屏：运行趋势（任务成功率、错误趋势）
- 第三屏：最近异常列表（可跳详情）
- 右侧固定快捷入口：新建智能体、发起对话、查看日志

## 16.2 智能体列表

- 顶部：筛选条（类型/状态/标签/搜索）
- 中部：表格 + 批量操作
- 右侧抽屉：新建/编辑表单（Stepper）
- 底部：分页与刷新时间

## 16.3 智能体详情

- 顶部：名称、状态、启用开关、运行按钮
- Tab:
  - 配置（表单）
  - 调试运行（输入+输出）
  - 调度任务（关联任务）
  - 运行日志（时间线）
- 右侧信息卡：资源绑定摘要、风险提示、最近错误码

## 16.4 对话控制台

- 左：会话历史
- 中：对话流（支持 markdown/mermaid/a2ui）
- 下：输入区 + 停止按钮 + 发送按钮
- 顶：当前系统上下文（模型、scheduler、channel）
- 消息气泡内可展开：工具调用轨迹

## 17. 状态与反馈设计（必须补齐）

每个页面必须定义 5 种状态：

1. **Loading**：骨架屏（而非纯文字 loading）
2. **Empty**：空状态插画+下一步 CTA
3. **Error**：错误原因 + 重试按钮 + 文档链接
4. **Success**：操作成功提示（可撤销更好）
5. **Partial**：部分成功/部分失败（常见于批量任务）

交互反馈规范：

- 提交按钮进入 loading 态，避免重复点击
- 保存后高亮“已生效范围”（例如“影响内部智能体运行”）
- 破坏性操作统一二次确认对话框（替换裸 `confirm`）

---

## 18. 响应式与可访问性

## 18.1 响应式

- 移动端默认隐藏复杂表格列，改为卡片列表
- 侧边栏在移动端抽屉化，保留最短操作路径
- 表单在移动端改单列，不出现横向滚动

## 18.2 可访问性（A11y）

- 所有图标按钮必须有 `aria-label`
- 键盘可达（Tab 顺序、Enter 提交、Esc 关闭）
- 颜色状态需配合文本，不依赖颜色单通道
- 错误信息需与输入字段绑定（`aria-describedby`）

---

## 19. 与当前代码的具体对齐建议

1. `chat/page.tsx`：保留当前高质量渲染能力，补“系统上下文栏 + 工具轨迹折叠”。  
2. `agents/page.tsx`：从“原始 JSON 表格”改为“业务字段表格 + 详情跳转”。  
3. `internal-agents/page.tsx`：并入 `agents` 详情的“调试运行”Tab，作为兼容入口保留。  
4. `scheduled-tasks/page.tsx`：加入 dry-run 与状态摘要卡。  
5. `gateway-logs/page.tsx`：增加结构化过滤（event/error_code/agent_name）与跳转链路。  
6. `config/page.tsx`：维持编辑器优势，增加“引用此 prompt 的智能体数量”侧栏信息。

## 20. 前端实施拆解（设计+研发协同）

### Sprint 1（系统感基础）

- 导航分组重构
- 系统总览页
- 全局 Header 状态灯

### Sprint 2（智能体闭环）

- 智能体列表重构
- 智能体详情页（配置+调试）
- 统一弹窗和反馈组件

- 任务调度向导
