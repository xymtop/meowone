# MeowOne AI 操作系统 — 技术方案 V1

---

## 一、架构哲学

MeowOne 的后端核心不是一个"调度系统"，而是一个 **Agent Loop 运行时**。

传统的 AI 应用架构通常是"管道式"的：用户输入 → 意图识别 → 路由 → 执行 → 返回。每一步都是独立的基础设施模块，由工程代码硬编码流程。这种架构的问题在于：流程是僵死的，每新增一种场景就需要新增一条管道。

MeowOne 采用完全不同的思路：**系统的核心是一个智能体，它在一个循环中自主思考和行动。** 意图识别、任务调度、工具调用、结果整合——这些不是独立的系统模块，而是智能体在循环中自然涌现的行为。

这和 Claude Code 的工作方式本质相同：

```
用户说了一句话
  → 智能体开始思考
  → "我应该直接回答，还是需要用工具？"
  → 决定调一个工具 → 拿到结果
  → "结果够了吗？还需要做什么？"
  → 决定再调另一个智能体 → 拿到结果
  → "现在我可以回复用户了"
  → 组织回复，结束本轮
```

**没有预设的流程，每一步都是智能体根据当前上下文自主决策的结果。**

## 二、核心架构概念

### 2.1 Agent Loop（智能体循环）

Agent Loop 是整个系统的心脏。它是一个 `Think → Act → Observe` 的持续循环：

```
                    ┌──────────────────────────┐
                    │       Agent Loop         │
                    │                          │
  User Input ──▶   │   ┌───────┐              │
                    │   │ Think │ LLM 推理     │
                    │   └───┬───┘              │
                    │       │                  │
                    │       ▼                  │
                    │   ┌───────┐              │
                    │   │  Act  │ 执行动作     │   ──▶ Capability 池
                    │   └───┬───┘              │       (工具 + 智能体)
                    │       │                  │
                    │       ▼                  │
                    │   ┌─────────┐            │
                    │   │ Observe │ 观察结果    │
                    │   └───┬─────┘            │
                    │       │                  │
                    │       ▼                  │
                    │   够了吗？               │
                    │    │      │              │
                    │   否      是             │
                    │    │      │              │
                    │    ▼      ▼              │
                    │  继续    输出给用户       │   ──▶ User Output
                    │  Loop                    │       (文本 / 卡片 / 混合)
                    │                          │
                    └──────────────────────────┘
```

**Think 阶段**：LLM 根据完整上下文（系统提示 + 对话历史 + 可用能力清单 + 上一步的 Observe 结果）进行推理，决定下一步做什么。

**Act 阶段**：执行 Think 的决策。可能是调用本地工具、调用远程智能体、向用户输出中间结果、或者请求用户补充信息。

**Observe 阶段**：将 Act 的结果写入上下文，供下一轮 Think 使用。

**终止条件**：LLM 判断任务已完成，或触发最大轮次限制、超时等安全阈值。

### 2.2 Capability（能力）

从 Agent Loop 的视角看，它能做的所有事情都是"能力"。工具和远程智能体没有本质区别：

```
┌─────────────────────────────────────────────────┐
│                Capability 池                     │
│                                                 │
│   本地工具 (Tools)            远程智能体 (Agents) │
│   ┌───────────────────┐     ┌─────────────────┐ │
│   │ card_builder      │     │ weather_agent   │ │
│   │ db_query          │     │ meeting_agent   │ │
│   │ file_reader       │     │ email_agent     │ │
│   │ calculator        │     │ search_agent    │ │
│   │ ...               │     │ ...             │ │
│   └───────────────────┘     └─────────────────┘ │
│                                                 │
│   对 Agent Loop 来说，两者都是：                   │
│     名称 + 描述 + 参数 → 调用 → 返回结果          │
└─────────────────────────────────────────────────┘
```

每个 Capability 对外暴露统一的描述格式，供 LLM 在 Think 阶段理解和选择：

- **名称**：唯一标识
- **描述**：自然语言说明这个能力能做什么
- **参数定义**：输入 schema
- **返回定义**：输出 schema

本地工具通过函数调用执行；远程智能体通过通信协议调用。这个差异被封装在 Capability 层内部，Agent Loop 不感知。

### 2.3 对等智能体模型

MeowOne 中所有智能体都是 **一等公民**，包括主智能体自身。它们的关系是队友，不是上下级。

```
              ┌─────────┐
              │  用户    │
              └────┬────┘
                   │
           ┌───────▼────────┐
           │ 主智能体 (Meow) │  ← 也是一个普通 Agent
           │                │     只是多了"调度"能力
           │  · Agent Loop  │     和用户接口
           │  · 本地工具     │
           │  · 调度能力     │
           └┬──────┬───────┬┘
            │      │       │
        ┌───▼┐  ┌──▼──┐ ┌─▼───┐
        │ B  │  │  C  │ │  D  │  ← 和主智能体同构
        └────┘  └─────┘ └─────┘     各自有自己的能力边界
```

关键原则：

- **同构**：每个智能体都有名称、描述、技能列表，都可以接收任务、返回结果
- **星型通信**：主智能体只管自己和每个被调用 Agent 之间的通信线路，不管 Agent 之间的互联
- **自主性**：被调度的智能体如何完成任务是它自己的事，主智能体只关心"给了什么输入、拿到什么输出"

### 2.4 通信协议抽象

主智能体与远程智能体之间的通信需要协议。默认采用 **A2A（Agent-to-Agent）协议**，但协议本身是可替换的。

```
Agent Loop
    │
    │  "我要调用 weather_agent"
    │
    ▼
┌─────────────────────────────────┐
│        协议适配层                │
│                                 │
│   ┌─────────┐                   │
│   │  A2A    │ ← 默认协议        │
│   ├─────────┤                   │
│   │  Local  │ ← 开发期直调      │
│   ├─────────┤                   │
│   │  ...    │ ← 未来扩展        │
│   └─────────┘                   │
│                                 │
│   统一接口:                      │
│     discover() → 获取能力描述    │
│     send()     → 发送任务       │
│     stream()   → 流式任务       │
│     cancel()   → 取消任务       │
└─────────────────────────────────┘
```

**A2A 协议核心映射：**

| A2A 概念 | MeowOne 对应 | 说明 |
|---|---|---|
| Agent Card | Capability 描述 | 每个 Agent 的能力自描述，通过 `/.well-known/agent.json` 暴露 |
| Task | 一次能力调用 | 有完整生命周期：submitted → working → completed / failed |
| Message | 调用的输入输出 | 包含 Parts（TextPart / DataPart） |
| Artifact | Agent 产出物 | 任务完成后的输出（文本、结构化数据等） |
| Streaming | SSE 流式推送 | 长任务的中间状态和最终结果 |

**为什么协议要可插拔：**

- V1 阶段本地开发用 `Local` 直调（零网络开销），生产用 `A2A`
- 未来可能接入 MCP 协议的工具型 Agent
- 企业场景可能需要私有协议
- 协议变化不应影响 Agent Loop 和业务逻辑

### 2.5 调度行为的涌现

传统方案会预定义 `DirectStrategy`、`SequentialStrategy` 等硬编码策略。MeowOne 不这样做。

调度策略是 **Agent Loop + LLM 推理自然涌现**的行为，而非预设的代码路径：

| 涌现行为 | LLM 的推理过程 |
|---|---|
| 直接回答 | "这个问题我自己就能答，不需要调任何能力" |
| 单次调用 | "我需要查一下天气，调 weather_agent" |
| 串行调用 | "先查天气，看结果再决定要不要创建会议" |
| 并发调用 | "查天气和查日程互不依赖，我同时调" |
| 交互式 | "我需要用户确认一下再继续，先返回一张操作卡片" |
| 错误恢复 | "weather_agent 失败了，我换种方式试试" |

**工程上需要做的不是硬编码策略，而是：**

1. **Prompt 引导**：通过 System Prompt 告诉 LLM 它有哪些能力、如何选择、何时并发、何时需要用户确认
2. **运行时支撑**：Loop 运行时要能支撑这些涌现行为——并发调用、挂起等待、流式输出、错误后继续 Loop
3. **安全边界**：最大 Loop 轮次、单次超时、总超时，防止无限循环

## 三、技术选型

| 层级 | 选型 | 理由 |
|---|---|---|
| 前端框架 | Next.js 15 (App Router) + TypeScript | SSR 支持、文件路由、React 生态成熟 |
| UI 样式 | Tailwind CSS + shadcn/ui | 原子化样式、高质量组件库 |
| 状态管理 | Zustand | 轻量、直觉式 API |
| 实时通信 | WebSocket + SSE | 双向交互 + 流式推送 |
| 后端框架 | Python FastAPI | 原生异步、自动 OpenAPI 文档 |
| LLM 调用 | 手写 HTTP 客户端（httpx） | 直接调 OpenAI 兼容 API，无框架依赖 |
| Agent 编排 | 手写 Agent Loop | 完全透明可控，零学习成本 |
| 数据库 | SQLite | 文件级数据库，零部署成本 |
| 缓存 | 进程内存字典 | V1 阶段够用，无需外部中间件 |
| 文件存储 | 本地文件系统 | 附件存目录即可（预留） |

**整个系统的外部依赖：Node.js + Python + 一个 LLM API Key。无需安装任何中间件。**

### 5.2 UI 协议系统

后端通过 JSON 协议描述卡片内容，前端根据协议动态渲染。这是 MeowOne 区别于普通聊天工具的核心差异。

**三种卡片类型：**

| 类型 | 用途 | 特点 |
|---|---|---|
| InfoCard | 展示结构化信息 | 只读，字段列表 |
| ActionCard | 引导用户操作 | 带按钮，点击触发下一步 |
| FormCard | 收集用户输入 | 带表单，提交后转化为对话 |

**卡片协议定义：**

```typescript
interface BaseCard {
  id: string;
  type: "info" | "action" | "form";
  title: string;
  icon?: string;
  status?: "loading" | "success" | "error";
}

interface InfoCard extends BaseCard {
  type: "info";
  fields: Array<{ label: string; value: string }>;
}

interface ActionCard extends BaseCard {
  type: "action";
  fields: Array<{ label: string; value: string }>;
  actions: Array<{
    id: string;
    label: string;
    style: "primary" | "secondary" | "danger";
    payload: Record<string, unknown>;
  }>;
}

interface FormCard extends BaseCard {
  type: "form";
  fields: Array<{
    name: string;
    label: string;
    type: "text" | "date" | "select" | "number";
    placeholder?: string;
    required?: boolean;
    options?: Array<{ label: string; value: string }>;
  }>;
  submitLabel: string;
}

type Card = InfoCard | ActionCard | FormCard;
```

**行为即对话**：用户在卡片上的操作（点击按钮、提交表单）会被转化为一条新消息重新进入 Agent Loop，驱动对话继续。

### 5.3 实时通信

采用 **SSE + WebSocket 双通道**：

- **SSE**：Agent Loop 的流式文本输出（单向、轻量、HTTP 兼容）
- **WebSocket**：双向交互（卡片状态更新、typing 指示、连接管理）

```
客户端 → 服务端:
  { event: "message",     data: { content, type } }
  { event: "card_action", data: { cardId, actionId, payload } }

服务端 → 客户端:
  { event: "delta",       data: { messageId, content, done } }
  { event: "card",        data: { messageId, card } }
  { event: "card_update", data: { cardId, status } }
  { event: "thinking",    data: { step, description } }
  { event: "error",       data: { code, message } }
```

`thinking` 事件是 Agent Loop 特有的：当主智能体在执行多步操作时，可以实时推送当前正在做什么（"正在查询天气..."、"正在创建会议..."），让用户感知到系统在工作。

### 6.3 Agent Loop 运行时详解

Loop 运行时的核心职责：

**1. 上下文构建**

每轮 Loop 开始时，组装完整的 LLM 上下文：

```
System Prompt（你是 MeowOne 主智能体，你有以下能力可以使用...）
+ 对话历史（用户和助手的历史消息）
+ 可用能力清单（每个 Capability 的名称和描述）
+ 本轮的中间结果（如果不是第一轮 Loop）
```

**2. LLM 调用与决策解析**

调用 LLM，解析它的输出：
- 如果输出是纯文本 → 流式推送给用户，Loop 结束
- 如果输出是工具/智能体调用请求 → 进入 Act 阶段
- 如果输出是多个并发调用请求 → 并发执行，全部完成后进入 Observe

**3. 执行与观察**

执行 Capability 调用，将结果写入上下文，进入下一轮 Think。

**4. 安全控制**

- 最大 Loop 轮次（默认 10 轮，可配置）
- 单次工具调用超时（默认 30s）
- 总执行时间上限（默认 120s）
- 达到限制时，LLM 被要求生成一个"总结当前进度"的回复

**5. 可观测性**

每轮 Loop 的 Think / Act / Observe 都记录到 `loop_logs` 表：
- 第几轮
- Think 的推理结果
- Act 调用了什么、耗时多少
- Observe 拿到了什么结果
- 最终是否完成

这是调试和优化 Agent 行为的关键数据。

### 6.4 Loop 的可移植性设计

Agent Loop 是 MeowOne 最核心的模块，它的实现必须 **与 FastAPI 完全解耦**。Loop 不应该知道自己运行在什么 Web 框架里、结果通过什么方式推送给用户。它只做一件事：接收输入，生成事件流。

**核心设计：Loop 是一个 AsyncGenerator，输出事件流**

```
run_loop(
    message,              # 用户消息
    context,              # 对话上下文
    capabilities,         # 注入: 能力注册表
    llm_client,           # 注入: LLM 客户端
    protocol_registry,    # 注入: 协议适配器
    config,               # 注入: Loop 参数（最大轮次、超时等）
) -> AsyncIterator[LoopEvent]
```

Loop 所有依赖通过参数注入，不硬编码任何具体实现。事件流中的事件类型：

```
ThinkingEvent     → Loop 正在思考 / 执行中间步骤
DeltaEvent        → LLM 流式文本片段
ToolCallEvent     → 准备调用某个 Capability
ToolResultEvent   → Capability 调用返回结果
CardEvent         → 生成了一张 UI 卡片
ErrorEvent        → 发生错误
DoneEvent         → Loop 结束
```

**为什么必须这样设计：**

不同消费场景对 Loop 事件流的消费方式完全不同：

| 消费者 | 消费方式 |
|---|---|
| **FastAPI SSE 端点** | 遍历事件流，序列化为 SSE 格式推送到浏览器 |
| **Python SDK** | 包装为 `meowone.chat()` 函数，开发者直接 `for event in chat(...)` |
| **CLI 工具** | 遍历事件流，打印到终端 |
| **其他平台集成** | 将事件序列化为平台要求的格式转发 |
| **单元测试** | 直接调用 `run_loop()`，断言事件序列 |

如果 Loop 和 FastAPI 耦合（比如在 Loop 内部直接写 WebSocket），那么要做 SDK 就得把 Web 层全部重构。而事件流设计下，**FastAPI 只是 Loop 的众多消费者之一**。

**SDK 封装示意：**

```
# 未来的 meowone-sdk

import meowone

client = meowone.Client(api_key="...", base_url="...")

# 同步用法
response = client.chat("帮我查明天天气")
print(response.text)

# 流式用法
for event in client.chat_stream("帮我查明天天气"):
    if event.type == "delta":
        print(event.content, end="")
    elif event.type == "card":
        print(event.card)
```

这个 SDK 的底层，就是对 `run_loop()` 事件流的封装（本地模式），或者对 SSE API 的封装（远程模式）。因为 Loop 的事件协议是统一的，两种模式对 SDK 使用者来说没有区别。

## 九、关键技术细节

### 9.1 流式输出

Agent Loop 中 LLM 的文本生成采用流式传输：

```
用户发送消息
  → 后端立即返回 SSE 连接
  → 推送 thinking 事件
  → Agent Loop 第 1 轮 Think: LLM 开始生成
  → 逐 token 推送 delta 事件
  → 如果 LLM 决定调用能力 → 推送 thinking 事件 ("正在执行...")
  → 能力返回结果 → Loop 继续
  → 最终文本/卡片 → 推送 card / delta 事件
  → 推送 done 事件
```

### 9.2 会话上下文管理

```
每次用户发送消息时:
  1. 从内存缓存读取上下文；缓存未命中则从 SQLite 加载
  2. 构建 LLM messages: system_prompt + 能力清单 + 历史消息 + 新消息
  3. Loop 执行过程中的中间结果也纳入上下文
  4. Loop 结束后，新消息持久化到 SQLite，更新内存缓存
  5. 超过上下文窗口时，对早期消息做摘要压缩
```

### 9.3 错误处理

```
能力调用失败时:
  → 失败结果作为 Observe 写入上下文
  → 下一轮 Think，LLM 看到失败信息，自主决定:
      · 换一种方式重试
      · 跳过这一步继续
      · 向用户解释情况
  → 这不是预设的降级策略，而是 LLM 根据上下文自然做出的反应

WebSocket 断线时:
  → 客户端自动重连（指数退避，最大 30s）
  → 重连后拉取离线期间的消息
  → 超过 5 次失败 → 提示用户检查网络
```

### 9.4 会话标题自动生成

```
用户发送第一条消息后:
  → 异步（不阻塞主 Loop）调用 LLM 生成 ≤ 20 字的标题
  → 更新 session.title
  → 通过 WebSocket 推送到前端 Sidebar
```
