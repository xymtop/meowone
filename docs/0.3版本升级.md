# MeowOne 0.3 版本升级方案 — 低侵入扩展（工具 / Agent 优先）

本文档面向后续 **尽量少改核心后端**、主要通过 **新增工具、新增远程 Agent、改配置** 来演进能力（含调度类能力）的路线。与「大改 `run_loop`、抽象 SchedulingPolicy」的方案相比，本方案 **优先控制代码变更面**，把可变部分外移到 **可插拔的能力单元**。

---

## 1. 原则（与 V1 技术方案一致处）

- **Agent Loop + 能力 = 工具** 仍是默认心智：主模型在循环里决定何时调用何种能力。
- **核心运行时**（`loop/runtime.py`、`CapabilityRegistry`、A2A/MCP 适配）尽量 **稳定**；不为了「理论上更优雅」频繁重构。
- **调度策略、优化算法、实验性流程** 优先以 **新工具** 或 **新远程 Agent** 交付，而不是改循环内核。

---

## 2. 为什么这样代价更小

| 做法 | 典型代价 |
|------|----------|
| 改 `run_loop`、拆 PhaseExecutor、多策略分支 | 回归范围大、前后端事件语义要对齐、发版耦合重 |
| **新增一个 BaseTool**（本地函数） | 单文件或少数文件，注册进 registry，测试局部 |
| **新增远程 Agent**（独立服务 + `agents.yaml` 一条） | 与核心解耦，可独立迭代、灰度、下线 |
| **改 YAML / 环境变量** | 无代码发版或配置热更新（视部署而定） |

调度算法（蜂群、启发式、规则引擎）若写在 **工具实现** 或 **专用 Agent 服务** 里，**替换算法 = 换实现或换 Agent**，主仓库不必跟着改调度框架。

---

## 3. 推荐架构观（不强制大重构）

```
用户 / 主模型
    │
    ▼
Agent Loop（尽量少动）
    │
    ├── 现有：LLM 选 tool_calls → 并行 execute
    │
    └── 扩展：多出来的能力全部是「普通工具 / 普通远程 Agent」
              例如：plan_with_swarm、submit_dag_phase、invoke_subagent …
```

**关键点**：复杂逻辑放在 **工具体** `execute()` 或 **远程 Agent 进程** 内；主循环仍只做「解析 tool_calls → 调 registry」，**不必**先抽象一层通用 Scheduler 接口才能做蜂群。

---

## 4. 调度 / 优化类需求怎么落地（按侵入度排序）

### 4.1 仅新增本地工具（最小）

- 在 `capability/tools/`（或同类目录）新增一个 **BaseTool**，例如：
  - 输入：任务列表、约束、可选 agent 列表（名字与 `agents.yaml` 一致）。
  - 内部：运行蜂群 / 其它算法，返回 **结构化计划**（JSON）。
- 主模型 **可选** 在下一轮按该 JSON 去调 `invoke_subagent` 或其它工具。  
  **代价**：若完全依赖模型「遵守计划」，执行可靠性依赖提示词与模型能力。

**适用**：探索期、内部试用、对执行顺序不敏感的任务。

### 4.2 工具 + 「执行计划」工具（仍少动核心）

- **工具 A**：只负责 **出计划**（与 4.1 相同）。
- **工具 B**：例如 `execute_scheduled_batch`，参数为「阶段列表 + 每阶段并行任务」；**在工具内部** 顺序调用现有 `CapabilityRegistry` 或封装好的 `RemoteA2AAgentCapability`（需注意异步与超时，避免阻塞事件循环过久）。

这样 **调度算法变更** 只改工具 A；**执行策略** 变更只改工具 B；主 `run_loop` **仍可不动**。

**适用**：希望计划与执行都服务端可控，又不愿改循环结构。

### 4.3 新增专用「调度 Agent」（远程）

- 把蜂群 / DAG 解析 / 成本模型放在 **独立 HTTP 服务**（与现有 `agents/code-writer` 同类）。
- 在 `.meowone/agents.yaml` **增加一条**，主模型像调用其它 specialist 一样调用。
- 主仓库改动：**仅配置 + 若需专用 tool 包装则薄封装**。

**适用**：算法迭代频繁、希望独立扩缩容或与 Python 版本隔离。

### 4.4 仅当上述仍不够时，再考虑「内核小改」

例如：统一超时、统一「工具内多段调用」的遥测、或 **一个** 可选入口「会话级 `scheduler_mode`」仅影响 **是否注入某几个工具到 OpenAI tools 列表**。仍应避免大而全的 `SchedulingPolicy` 分层，除非团队明确要长期维护多套一等公民策略。

---

## 5. 配置与元数据（优先配置文件）

- **Agent 列表与描述**：继续以 `.meowone/agents.yaml`（或 `AGENTS_CONFIG_PATH`）为主；需要给调度算法用的字段（预估耗时、成本权重、标签）可 **在同一文件扩展**，由 **工具或远程 Agent 自己读取**（或通过 API 下发），**不必**强行并进核心 `agents_config` 的必选 schema。
- **MCP**：若调度相关能力走 MCP，则 **新增 MCP server 或工具描述** 即可，与主循环解耦。

---

## 6. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 工具越来越「胖」，重复调用 registry 逻辑 | 抽 **私有模块**（非对外 API）供多个工具 import，仍不强制改 `run_loop` |
| 模型不遵守优化器给出的计划 | 4.2 的「执行计划」工具在服务端强制执行；或缩小主模型职责（系统提示里写明先调执行工具） |
| 工具内串行调用多 agent 变慢 | 在工具内对无依赖子任务用 `asyncio.gather`；或拆到远程 Agent 内并行 |

---

## 7. 实施顺序建议（与侵入度一致）

1. **文档与约定**：在团队内固定「调度类能力 = 工具或远程 Agent」，默认不改 Loop。
2. **第一个落地**：选 **4.1 或 4.3** 验证闭环（蜂群只产出计划即可）。
3. **需要可靠执行时**：加 **4.2** 的执行工具，避免依赖模型逐步点 subagent。
4. **仅当产品明确要求「会话级切换多种内核调度器」**：再评估是否引入更小范围的内核开关（仍优先配置 + 工具列表注入）。

---

## 8. 与《技术方案 V1》的关系

产品叙事仍是 **Agent Loop 为核心**；0.3 在工程上选择 **把「调度」当作一类普通能力（工具/Agent）挂载**，而不是先把运行时拆成多层框架。这与「复杂编排用额外能力扩展」的叙述 **兼容**，且 **更符合「少动代码、多扩展工具与 agent」** 的目标。
