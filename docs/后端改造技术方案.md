# 后端改造技术方案

## 1. 背景与目标

当前后端已经具备以下基础能力：

- 统一会话与消息链路（`/api/sessions/*`、`/api/gateway/turn`、SSE）
- 可复用的 Loop 运行时（`run_loop`）
- 智能体数据管理（`agents` 表，internal/external）
- 能力资源数据库化（`prompts`、`mcp_servers`、`skills`）
- 外部 A2A 工具化调用（`RemoteA2AAgentCapability`）

但当前存在关键问题：

1. 内部智能体“配置存在但运行时约束不强”：`mcp_servers`、`agent_skills` 主要以提示方式注入，缺少严格执行边界。  
2. 智能体缺少统一抽象：内部/外部调用路径分散，缺少统一的 `AgentRuntimePlan` 组装层。  
3. 调度的是“会话执行模式”，不是“智能体执行模型”：无法清晰表达“某个 agent 如何被装配并被调用”。

本方案目标：

- 建立统一的 Agent 抽象与运行时装配机制。
- 内部智能体复用 loop，不引入 scheduler 算法依赖。
- 外部智能体仅支持 A2A（优先），并固化协议配置与调用契约。
- 将 prompt/mcp/skill/tool policy 从“软提示”升级为“可验证约束”。

---

## 2. 改造原则

1. **不推翻现有主链路**：保持 `ConversationTurnService -> scheduler -> loop` 不变。  
2. **智能体抽象与会话调度解耦**：Agent 是“可执行单元”，Scheduler 是“任务执行策略”。  
3. **配置与执行分离**：数据库存静态定义，运行时生成动态计划。  
4. **渐进式落地**：先补齐装配层与约束，再逐步统一 API 与前端调用。  
5. **A2A 优先单协议**：外部智能体先固定 A2A，协议字段可预留扩展但不实现多协议。

---

## 3. 目标架构

```text
AgentDefinition (DB static)
  ├─ identity: name/type/enabled
  ├─ resources: prompt_ref/mcp_bindings/skill_bindings
  ├─ policy: allow_tools/deny_tools/limits
  └─ endpoint: protocol(a2a)/base_url/timeout/auth

          │ resolve(agent_name, invocation_context)
          ▼
AgentRuntimePlan (runtime dynamic)
  ├─ resolved_system_prompt
  ├─ resolved_capabilities (strictly filtered)
  ├─ resolved_limits
  └─ transport: internal_loop | external_a2a

          │ execute(plan)
          ▼
AgentDispatcher
  ├─ invoke_internal (run_loop)
  └─ invoke_external (RemoteA2A)
```

## 4. 数据模型改造

### 4.1 `agents` 表字段规范化（兼容现有）

现有字段可继续使用，但建议语义升级：

- `system_prompt`：改为“agent prompt overlay”（保留）
- `agent_skills`：绑定 skill 名称（保留，变为强约束输入来源）
- `mcp_servers`：绑定可用 MCP server 名称（保留，变为强约束）
- `allow_tools`/`deny_tools`：工具策略（保留）
- `max_rounds/max_tool_phases/timeout_seconds`：执行限额（保留）
- `base_url`：external endpoint（保留）

建议新增字段（phase-2，可先在 `extra_json` 过渡）：

- `protocol`：`internal_loop | a2a`（external 固定 `a2a`）
- `prompt_key`：引用 `prompts.prompt_key`（比直接存长文本更易治理）
- `metadata_json`：扩展位（如重试策略、超时覆盖、owner 等）

### 4.2 引用关系建议

- `agent.prompt_key -> prompts.prompt_key`
- `agent.agent_skills[*] -> skills.name`
- `agent.mcp_servers[*] -> mcp_servers.name`

说明：短期可在运行时校验，不强制 DB 外键；中期可增加“发布校验接口”保障一致性。

## 5. 核心模块改造

## 5.1 新增 `agent_definition.py`

定义统一领域模型：

- `AgentDefinition`
- `AgentToolPolicy`
- `AgentLimits`
- `AgentEndpoint`
- `AgentResources`

职责：

- 把 DB row 映射为强类型对象。
- 提供 `validate()`（字段完整性、协议合法性、资源存在性）。

## 5.2 新增 `agent_runtime_plan.py`

定义运行时计划：

- `resolved_system_prompt`
- `resolved_capabilities`
- `resolved_limits`
- `execution_transport`

## 5.3 新增 `agent_plan_builder.py`

职责：`AgentDefinition + InvocationContext -> AgentRuntimePlan`

关键行为：

1. 解析 prompt：加载 `prompt_key` 内容 + `system_prompt` 叠加。  
2. 加载 skill：从 DB 读取 skill body，并拼接为可控格式注入。  
3. 解析 MCP 绑定：生成 agent 级可用 server 集。  
4. 工具裁剪：基于 registry + allow/deny + mcp 绑定做最终过滤。  
5. 生成 limits：请求参数覆盖 agent 默认值（有限覆盖）。

## 5.4 新增 `agent_dispatcher.py`

统一执行入口：

- `invoke(agent_name, task, history, context) -> AgentExecutionResult`

执行分派：

- `internal` -> 构造 `LoopRunInput` 调 `run_loop`
- `external+a2a` -> 调 `RemoteA2ATransport`（可复用现有 `RemoteA2AAgentCapability` 逻辑）

统一返回：

- `ok`
- `agent_name`
- `agent_type`
- `output`
- `duration_ms`
- `loop_rounds`（internal）
- `error`

## 6. 内部智能体（重点）改造细则

### 6.1 目标

让内部智能体真正“可配置、可约束、可复用 loop”，并可稳定被调度调用。

### 6.2 关键改造点

1. **Prompt 从“字符串字段”升级为“引用 + overlay”**
   - 优先加载 `prompt_key` 对应内容，再拼接 `system_prompt`。
2. **Skill 从“名称提示”升级为“正文注入”**
   - 读取 `skills.body`，纳入 `extra_system` 的受控段落。
3. **MCP 从“偏好”升级为“白名单约束”**
   - 限制 `call_mcp_tool` 的 `server` 参数必须在绑定列表中。
4. **工具策略收敛**
   - `allow_tools` + `deny_tools` 在运行前即裁剪 registry（不是运行中放任模型试错）。
5. **Limits 固化**
   - agent 定义为默认，调用参数只能在可控范围内覆盖。

### 6.3 对现有代码影响

- `internal_factory.invoke()`：不再手工拼字符串，改为调用 `plan_builder + dispatcher`。
- `invoke_internal_agent` tool 和 `/api/internal-agents/{name}/invoke`：统一走 dispatcher。

---

## 7. 外部智能体（A2A）改造细则

### 7.1 目标

外部智能体配置最小但完整：协议明确、地址明确、调用稳定。

### 7.2 建议模型

- `agent_type = external`
- `protocol = a2a`（固定）
- `base_url = https://...`
- 可选 `metadata_json`: `timeout_seconds`, `retry`, `auth_type`

### 7.3 执行要求

1. 启动注册 capability 时，校验 external agent 协议为 a2a。  
2. 调用时统一走 A2A transport，输出标准化错误码：
   - `A2A_UNREACHABLE`
   - `A2A_PROTOCOL_ERROR`
   - `A2A_TIMEOUT`
3. tool 层错误不要抛异常中断 loop，返回可消费错误文本并附结构化码。

### 7.4 API 要求

- `POST /api/agents/external` 增加 `protocol` 字段（默认 `a2a`）
- 非 `a2a` 直接 400（当前阶段）

---

## 8. API 改造建议

## 8.1 保持兼容接口

- `/api/agents`、`/api/internal-agents`、`invoke_internal_agent` tool 保持可用。

## 8.2 新增统一执行接口（推荐）

- `POST /api/agent-executions`
  - 入参：`agent_name`, `task`, `history`, `overrides`
  - 出参：统一 `AgentExecutionResult`

收益：

- 前端/定时任务/工具调用都能走同一执行面。
- 与内部/外部实现解耦。

---

## 9. 与调度器的关系

你的要求是“内部智能体单纯复用 loop，不带调度算法”，建议明确：

1. **主会话调度器继续存在**（direct/master_slave/swarm）。  
2. **内部智能体执行固定 direct-loop**。  
3. 如果主智能体想调用内部智能体，它只是一次 tool delegation，不继承主会话 scheduler。  

这样语义清晰、复杂度最低。

---

## 10. 观测与可运维性

建议新增：

- `agent_execution_logs`（或扩展 `loop_logs`）
  - `execution_id`, `agent_name`, `agent_type`, `status`, `duration_ms`, `error_code`
- 统一 tracing id（gateway 请求与 agent 执行串联）
- 对 A2A 调用记录 endpoint/耗时/状态码（避免黑盒）

## 11. 迁移与实施计划

## Phase 1（1-2 天）：抽象与装配层

- 新增 `AgentDefinition` / `AgentRuntimePlan` / `AgentPlanBuilder`
- `internal_factory.invoke` 接入 plan builder
- skill/mcp/tool-policy 在运行时强约束生效

验收：

- 内部 agent 在配置 MCP 白名单后，无法访问未授权 server
- skill 正文确实参与 system prompt 组装

## Phase 2（1 天）：统一执行层

- 新增 `AgentDispatcher`
- `/api/internal-agents/*/invoke` 与 tool `invoke_internal_agent` 改走 dispatcher
- 新增可选 `/api/agent-executions`

验收：

- internal/external 输出结构一致
- 错误码稳定可识别

## Phase 3（1 天）：外部 A2A 规范化

- external agent 增加 `protocol` 字段
- 启动注册时协议校验
- A2A 错误码和重试策略落地

验收：

- 非 a2a 配置无法保存
- A2A 不可达时系统可回退并可观测

## Phase 4（0.5-1 天）：观测补齐

- 增加 agent execution log
- gateway 日志中关联 agent execution id

## 12. 风险与注意事项

1. **Prompt 膨胀风险**：skill + prompt 叠加后 token 激增，需长度预算和截断策略。  
2. **能力误授权风险**：MCP/工具必须默认 deny，明确 allow。  
3. **兼容风险**：前端已有 `/api/internal-agents` 与 `/api/agents` 双轨，迁移需保持兼容期。  
4. **外部依赖风险**：A2A SDK 与 Python 版本耦合，需在部署文档中明确最低版本。

7. 新增（可选）`/api/agent-executions`。

   - internal agent 资源装配测试（prompt/skill/mcp/tool policy）
   - A2A 不可达/超时测试
   - API 兼容性回归测试

---

## 14. 预期结果

改造完成后，系统将具备：

- 可统一描述和执行的 Agent 模型（内部/外部一致）
- 内部 agent 的强约束执行（不是“提示式约定”）
- 外部 agent 的 A2A 协议化治理
- 前端和调度层可稳定消费的一致执行结果结构

这将为后续的“多 agent 编排、任务编排可视化、权限治理”提供稳固底座。
