# MeowOne 版本升级 v2 — 技术方案（后端）

## 1. 文档目标

本文档定义 MeowOne v2 的后端技术改造方案，确保：

- 前端零改造（保持现有渲染与交互协议）
- 后端形成可扩展分层架构
- 调度核心（Core Loop）支持多种调度算法扩展
- 内部智能体能力可工厂化创建与调用
- 外部智能体支持 A2A 协议接入与自动发现介绍信息

---

## 2. 约束与原则

### 2.1 约束

- **前端不改动**：沿用现有 SSE 事件（`delta` / `thinking` / `tool_call` / `tool_result` / `card` / `done`）。
- **兼容现网行为**：默认行为与 v1/v0.3 一致，新增能力通过后端配置与策略启用。
- **渐进演进**：优先新增层与模块，不做一次性推翻式重构。

### 2.2 原则

- **分层清晰**：通道适配、SDK、调度、智能体执行职责隔离。
- **策略可插拔**：调度算法通过统一接口注册，不侵入 API 层。
- **能力工厂化**：内部 Agent 支持“上下文打包 -> 快速实例化 -> 可复用调用”。
- **协议标准化**：外部 Agent 以 A2A 为统一接入协议。

---

## 3. 目标分层架构

> 前端（渲染） -> 网关（API/渠道） -> 核心 SDK -> 调度核心 Core Loop -> Agent 执行层

### 3.1 分层职责

1. **渲染层（Frontend，保持不变）**
   - 接收 SSE 流并渲染文本、卡片、状态
   - 不感知调度算法与 Agent 来源（内部/外部）

2. **网关层（Gateway / API / Channel Adapter）**
   - 统一入口：`/api/sessions/{id}/chat`、`/api/gateway/turn`、`/v1/chat/completions`
   - 负责会话映射、渠道会话映射、能力白名单/黑名单、鉴权等

3. **核心 SDK 层（Core SDK）**
   - 提供统一编程接口：`run_turn(...)`、`invoke_agent(...)`、`select_scheduler(...)`
   - 向上屏蔽调度细节，向下封装能力注册与调用协议

4. **调度核心层（Core Loop）**
   - 承载任务理解、计划生成、工具/Agent 调用编排、重试与收敛
   - 通过 `SchedulerStrategy` 接口扩展不同调度算法

5. **Agent 执行层**
   - **内部 Agent**：由 Agent Factory 根据上下文创建
   - **外部 Agent**：通过 A2A 注册表发现并调用

---

## 4. 当前代码映射（现状）

当前后端已具备以下基础：

- 统一循环执行：`backend/app/loop/runtime.py`（多轮 + 并行 tool_calls）
- 编排说明锚点：`backend/app/loop/orchestration.py`
- 网关会话服务：`backend/app/gateway/turn_service.py`
- 子智能体调用工具：`backend/app/capability/tools/subagent_tool.py`
- 外部 A2A 能力封装：`backend/app/capability/tools/remote_a2a_agent.py`
- 外部 Agent 配置加载：`backend/app/agents_config.py`

v2 将在此基础上补齐“分层清晰 + 策略可扩展 + 内外智能体统一调度”。

## 5. v2 关键改造设计

### 5.1 网关层（API / 渠道端）

#### 目标

- 保持前端协议不变
- API 层只做“请求编排与上下文组装”，不承载调度策略逻辑

#### 改造点

- 在 `gateway/turn_service` 侧新增 `DispatchContext`（会话、渠道、租户、能力过滤、限制参数）。
- 网关层只调用 SDK：`sdk.run_turn(dispatch_context)`。
- 渠道差异（如飞书）继续放在 adapter，避免侵入 core loop。

---

### 5.2 核心 SDK 层

#### 目标

- 为上层网关提供稳定接口
- 为 core loop 与 agent 执行层提供依赖注入入口

#### 设计

新增（或扩展）SDK核心对象：

- `CoreSDK`
  - `run_turn(context) -> AsyncIterator[LoopEvent]`
  - `create_internal_agent(spec) -> InternalAgentHandle`
  - `invoke_external_agent(ref, task) -> AgentResult`
  - `register_scheduler(name, strategy)`

- `RuntimeContainer`
  - 装配 capability registry、scheduler registry、agent factory、a2a registry

#### 价值

- API 层与调度策略解耦
- 后续 CLI、批处理、Webhook 等新入口可直接复用 SDK

---

### 5.3 调度核心（Core Loop）可扩展设计

#### 目标

支持不同调度算法扩展（定向 / 主从 / 蜂群 / DAG / 规则引擎等）。

#### 核心接口

```python
class SchedulerStrategy(Protocol):
    name: str
    async def plan(self, request: ScheduleRequest) -> ExecutionPlan: ...
    async def execute(self, plan: ExecutionPlan, runtime: ExecutionRuntime) -> ScheduleResult: ...
```

```python
class SchedulerRegistry:
    def register(self, strategy: SchedulerStrategy) -> None: ...
    def resolve(self, mode: str) -> SchedulerStrategy: ...
```

#### 默认策略集（v2）

- `direct`：单 Agent / 单工具快速执行
- `master_slave`：主智能体拆解 + 子智能体执行
- `swarm`：并行探索 + 结果收敛（评分函数可配置）

#### 运行机制

- `core loop` 根据任务特征/配置选择策略
- 策略输出统一 `ExecutionPlan`，便于可观测与回放
- 保持现有 `run_loop` 的 streaming 语义；先以“策略内编排 + 统一事件转换”渐进接入

---

### 5.4 内部智能体 Loop（参考 Claude Code 思路）

#### 目标

抽离内部智能体 loop 层，封装“按上下文快速创建和调用智能体”的能力。

#### Agent 上下文模型

`AgentContextBundle`（可序列化）：

- `system_prompt`
- `mcp_servers`
- `agent_skills`
- `tools`
- `limits`（max_rounds / timeout / tool policy）
- `memory_scope`（session / task / ephemeral）

#### 核心组件

1. `InternalAgentFactory`
   - 输入 `AgentContextBundle + AgentSpec`
   - 输出可执行 `InternalAgentRunner`

2. `InternalAgentRunner`
   - 独立 loop 生命周期（plan -> act -> observe -> reflect）
   - 对外暴露统一 `run(task)` 接口
   - 结果标准化为 `AgentExecutionResult`

3. `InternalAgentPool`（可选）
   - 支持短时复用实例，降低重复构建成本

#### 与当前能力的关系

- 保留现有 `invoke_subagent` 工具语义
- 将“是否内部 / 外部”从调用方屏蔽，由调度器决定最优执行路径

---

### 5.5 外部智能体（A2A 协议调度）

#### 目标

- 支持通过配置动态接入外部 Agent
- 自动获取 agent 介绍信息（能力、输入输出描述）

#### 设计

1. **配置模型扩展**
   - 基于现有 `.meowone/agents.yaml` 增加字段：
     - `discovery_mode`（static / auto）
     - `agent_card_url`（可选）
     - `tags` / `cost_weight` / `latency_weight`
     - `auth`（token / header ref）

2. **自动发现**
   - 启动或定时刷新时访问 `/.well-known/agent-card.json`
   - 拉取并缓存：
     - agent name / description
     - capabilities / input schema / output schema
     - endpoint metadata

3. **A2A 调度注册**
   - `ExternalAgentRegistry` 按健康状态维护可用 Agent 列表
   - 调度器根据策略选择外部 Agent（成本、延迟、可靠性、标签匹配）

4. **容错机制**
   - 超时、重试、熔断、降级到内部 Agent 或本地工具
   - 错误结构统一，回传 core loop 可解释原因

## 6. 数据结构与配置建议

### 6.1 调度策略配置（新增）

`.meowone/scheduler.yaml`（示例）

```yaml
default_mode: master_slave
modes:
  direct:
    enabled: true
  master_slave:
    enabled: true
    max_subtasks: 6
  swarm:
    enabled: true
    fanout: 4
    reducer: score_vote
routing:
  by_channel:
    feishu: master_slave
    web: direct
  by_task_tag:
    research: swarm
```

### 6.2 外部 Agent 配置（扩展现有 agents.yaml）

```yaml
agents:
  - tool_name: code_writer
    base_url: http://127.0.0.1:8001
    enabled: true
    discovery_mode: auto
    tags: [coding, refactor]
    cost_weight: 0.7
    latency_weight: 0.3
```

## 7. 端到端时序（v2）

1. 用户请求进入网关（web/feishu 等）
2. 网关构建 `DispatchContext` 并调用 `CoreSDK.run_turn`
3. CoreSDK 选择调度策略（默认 + 动态路由）
4. 调度核心生成计划并执行：
   - 内部 Agent：Factory 创建 Runner 并执行 loop
   - 外部 Agent：A2A Registry 选择目标并调用
5. 汇总结果 -> 转换为现有 SSE 事件 -> 前端渲染（无需改动）
6. 过程数据写入会话与观测日志，支持回放与优化

---

## 8. 可观测性与治理

- **链路追踪**：`session_id / turn_id / scheduler_mode / agent_id / tool_call_id`
- **核心指标**：
  - 调度成功率
  - 平均完成时长
  - 外部 Agent 可用率
  - 重试率 / 降级率
- **审计回放**：
  - 计划快照（ExecutionPlan）
  - 执行轨迹（每阶段输入输出摘要）
  - 失败原因分类（超时/不可用/解析失败）

## 9. 里程碑（技术实施）

### M1：分层与 SDK 落地（1-2 周）

- 网关 -> SDK 调用链改造完成
- 引入 `SchedulerRegistry` 与 `direct` 策略
- 保证现有行为不回归

### M2：调度可插拔（1-2 周）

- 上线 `master_slave` / `swarm` 策略
- 增加 `scheduler.yaml` 配置与动态路由
- 引入计划与执行结构化日志

### M3：内外智能体统一调度（2 周）

- `InternalAgentFactory` + Runner 落地
- A2A 自动发现（agent-card）与注册缓存
- 失败降级与熔断机制完善

### M4：稳定性与治理增强（1 周）

- 指标看板与告警
- 压测与容错演练
- 版本验收与灰度策略

---

## 10. 验收标准（技术视角）

- 前端零改造即可跑通 v2 全链路
- 可通过配置新增/切换调度算法，无需改 API 层代码
- 内部 Agent 可由上下文信息快速创建并执行
- 外部 A2A Agent 支持自动发现介绍信息并纳入调度
- 关键链路可追踪、可回放、可降级

---

## 11. 风险与应对

- **风险：策略复杂度上升导致调试困难**  
  应对：统一 `ExecutionPlan` 与标准日志字段，先少策略后扩展。

- **风险：外部 Agent 稳定性参差不齐**  
  应对：健康检查 + 熔断 + 超时 + 回退策略。

- **风险：内部 Agent 上下文过重导致成本高**  
  应对：上下文预算控制、技能裁剪、按任务动态注入。

- **风险：兼容性回归影响现有用户**  
  应对：默认 `direct` 策略、灰度开关、渠道分批启用。

---

## 12. 结论

v2 技术升级的核心是：在不改前端的前提下，把后端从“单循环执行”升级为“分层、可扩展、可治理的智能体调度平台”。

最终实现：

- 架构上：网关、SDK、调度核心、Agent 执行层解耦
- 能力上：内部 Agent 工厂化、外部 Agent A2A 化
- 演进上：调度算法可插拔，平台可持续扩展
